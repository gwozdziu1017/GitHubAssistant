# GitHubAssistant — Project Bootstrapper CLI

## Overview
A CLI script that takes a `*_SPEC.md` file as input, auto-generates epics and detailed user stories, creates a local and remote GitHub repository, sets up a project board, and populates it with issues — ready for Claude Code to pick up and work on autonomously.

---

## CLI Usage
```bash
python githubassistant.py WISE-MAN_SPEC.md
```

Input must follow `*_SPEC.md` naming convention.

---

## Flow
1. Read and parse `*_SPEC.md` file
2. Claude analyses spec and generates:
   - Epics (used as GitHub labels)
   - Detailed user stories per epic
3. Display full plan — epics, labels, all user stories
4. **Wait for user acceptance before proceeding**
5. Create local Git repository
6. Set default branch to `master`
7. Create `dev` branch
8. Create remote private GitHub repository
9. Connect local to remote
10. Push `master` and `dev` branches
11. Create GitHub Project board
12. Create board columns: To Do, In Progress, Blocked, Done
13. Create GitHub issues from user stories
14. Apply epic labels to each issue
15. Assign all issues to To Do column
16. Display completion summary

---

## Epic Generation
- Claude reads the spec and automatically identifies logical groupings
- Each grouping becomes an epic and a GitHub label
- Example epics for Wise Man:
  - `setup` — project structure, dependencies, environment
  - `search` — Brave and Google Search integration
  - `analysis` — Claude verdict and credibility analysis
  - `output` — Rich terminal formatting
  - `cli` — argument handling and user interaction

---

## User Story Format
Each issue created on GitHub follows this structure:

```
Title: [Epic] Short descriptive title

Description:
Clear explanation of what needs to be built, why it exists,
and how it fits into the overall project.

Steps to cover:
1. Detailed implementation step
2. Detailed implementation step
3. Add error handling for X case
4. Add to .env if credentials needed
5. Test with Y input
...

Epic: search
Label: search
```

Stories are as detailed as the spec requires — no artificial brevity. Claude Code must be able to pick up any issue and work on it without needing clarification.

## Key Design Decisions
- **Acceptance before action** — full plan shown, nothing touches GitHub until user confirms
- **Auto-generated epics** — no manual configuration needed, Claude reads the spec
- **Detailed user stories** — Claude Code can work autonomously without clarification
- **Verbose by default** — user always knows what the script is doing
- **Private by default** — repositories created as private unless overridden in `.env`
- **Follows global CLAUDE.md** — master instead of main, master + dev branches always created

## Notes
- `gh` CLI must be authenticated before running — script checks this first and exits with clear error if not
- Repository name asked interactively — script prompts user for repo name before proceeding
- Script creates local repo in current working directory
- If repo already exists locally or remotely, script warns and asks before proceeding
